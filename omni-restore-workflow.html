The vendor's backup/restore moves the data — but only a vanilla system restores cleanly. Custom code, UDFs, and off-label field usage all leave gaps the restore itself will never report. This page walks the team through how the Fingerprint application finds those gaps, proves the fixes, and certifies the result — with no AI in the loop: every verdict comes from the deterministic engine.

ROLE 1

Extraction spec

Defines the logical order and exact layout to unload BOTH systems. Extract programs are stamped from it — the matrix row is the MOVE statement.

ROLE 2

Expectation model

Every position's intended type, domain and required-ness. Data that violates it — before or after restore — is a finding.

ROLE 3

Correction vocabulary

In Omni, fixes ARE transactions (T613, T813…). The same layouts that detect a gap define the corrective card deck that repairs it.

GUARDRAILS

Why the team (and the client) can trust the output

Deterministic by construction

Same extracts + same fingerprint version = byte-identical results. Money is exact decimal, never floating point. No clock, no randomness in any rule.

No AI in the loop

Extract scripts come from templates, verdicts from rules, narratives from the analyst. The sign-off narrative is written by the conversion lead, not generated.

It gets smarter anyway

Every confirm/false-positive review moves the failure-mode probabilities via the learning loop, and new gap patterns the team discovers are authored in by hand. Wave two starts with wave one's lessons.
